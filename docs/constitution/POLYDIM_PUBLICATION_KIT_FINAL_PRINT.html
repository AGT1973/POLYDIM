1. LinkedIn — Post Principal

Versión corta (post)

🚀 Publicamos POLYDIM: el primer lenguaje de programación diseñado para operar directamente en el espacio geométrico de los transformers.

El problema: los modelos de IA “piensan” en miles de dimensiones simultáneas, pero los forzamos a comunicarse por texto secuencial. Cada vez que un agente IA serializa su estado a JSON y otro lo parsea, se pierde más del 99.9% de la información semántica.

Interlat (AAAI-2026) midió esto: un hidden state contiene ~40,000 bits de información. Un token de texto: ~15 bits.

POLYDIM cierra esa brecha con 4 primitivas algebraicas:

  • COMPOSE — composición de transformaciones geométricas
  • MIX — superposición continua (reemplaza el if/else binario)
  • FIXPOINT — convergencia iterativa (reemplaza los loops)
  • PROJECT — proyección a ejecutores: Rust, Flutter, SQL, Graph

El programa no es texto. El programa ES la transformación tensorial.

📐 4 teoremas probados incondicionalmente 🦀 VM en Rust (sin dependencias externas) 🐍 Bootstrap Python con 29/29 tests 📄 Paper completo: github.com/AGT1973/POLYDIM

El lenguaje ensamblador de la era del transformer está aquí.

Versión larga (artículo LinkedIn)

Título: “POLYDIM: Por qué los agentes IA necesitan un lenguaje propio (y cómo lo construimos)”

Intro: Los CPUs de los años 50 ya hacían aritmética binaria, pero no existía un lenguaje para programarlos deliberadamente. El ensamblador cerró esa brecha. Hoy, los transformers ya operan en geometría multidimensional, pero no existe un lenguaje para programar esa geometría deliberadamente. POLYDIM cierra esa brecha.

El proyecto incluye: - Especificación formal completa (28 specs en /spec) - VM en Rust (polydim_core.rs, sin dependencias externas) - Bootstrap Python (29/29 tests, +64.5% mejora semántica sobre baseline) - Semántica denotacional en Para(Vect)

Paper: github.com/AGT1973/POLYDIM Código: github.com/AGT1973/POLYDIM

2. Twitter/X — Hilo de 8 Tweets

Tweet 1/8: 🧵 Presentamos POLYDIM: un lenguaje de programación cuya unidad fundamental no es una instrucción — es una transformación geométrica T: R^N → R^N.

El programa no corre SOBRE el transformer. El programa ES el transformer.

Tweet 2/8: El problema: los LLMs “piensan” en 10,000 dimensiones simultáneas. Los forzamos a hablar por texto secuencial.

Interlat (AAAI-2026): un hidden state = ~40,000 bits. Un token = ~15 bits.

Tiramos el 99.9% de la información semántica en cada serialización.

Tweet 3/8: POLYDIM tiene 4 primitivas algebraicas:

  • COMPOSE(T₁, T₂) = T₂ ∘ T₁
  • MIX(α, T₁, β, T₂) = α·T₁ + β·T₂ — el if/else geométrico
  • FIXPOINT(T, ε) — el loop geométrico
  • PROJECT(T, E) — proyección a Rust, Flutter, SQL…

Tweet 4/8: 4 teoremas probados incondicionalmente:

  • T1: COMPOSE es asociativo
  • T2: MIX preserva linealidad
  • T3: PROJECT es un funtor F: G → E (para COMPILE, RENDER y EXPORT)
  • T4: FIXPOINT converge (Banach)

Tweet 5/8: El resultado empírico:

  • 29/29 tests en el bootstrap Python
  • align_score = 0.9993
  • +64.5% mejora semántica vs baseline determinístico
  • Rust VM sin dependencias externas

Tweet 6/8: La analogía correcta:

El ensamblador no inventó el transistor. Lo hizo programable.

POLYDIM no inventa el transformer. Lo hace programable — por primera vez, con semántica formal.

Tweet 7/8: POLYDIM está solo en el espacio formal+geométrico+continuo.

  • Interlat/LatentMAS: empíricos, sin spec formal
  • NeSyCat: formal, pero discreto
