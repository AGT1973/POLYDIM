POLYDIM — Constitución V2

Documento raíz. Inmutable salvo proceso formal de enmienda. En contradicción con cualquier otro documento: este prevalece.

ARTÍCULO I — NATURALEZA

POLYDIM es un lenguaje de programación nativo del paradigma transformer.

Su unidad fundamental es una transformación: T : R^N → R^N

Un programa POLYDIM es una composición de transformaciones sobre un estado vectorial en R^N. POLYDIM no tiene instrucciones. No tiene pasos. Tiene geometría.

ARTÍCULO II — PROPÓSITO

  1. Comunicación AI↔︎AI — transformaciones T, no mensajes de texto
  2. Comunicación AI↔︎Humano — PROJECT(estado, DIM_FLUTTER), humano dentro del loop
  3. Ejecución multiplataforma — un único .polydim, cualquier plataforma

ARTÍCULO III — LO QUE NO EXISTE EN POLYDIM

Concepto Por qué no existe
Variables con nombre Hay posiciones en R^N con identidad geométrica
Funciones f(x) Hay transformaciones T: R^N → R^N que se componen
Loop for/while Hay convergencia a punto fijo: T^n(v) hasta estabilidad
Condicional if/else Hay gate continuo: α·T₁ + (1-α)·T₂
Tipo declarado El tipo emerge de la dimensión de observación
Instrucción secuencial Hay composición algebraica T₂ ∘ T₁
Mensaje de texto entre AIs Hay transmisión de T

ARTÍCULO IV — LAS CUATRO PRIMITIVAS

ATTEND ( Q, K, V, estado )

ATTEND(Q, K, V, s) = softmax( Q·Kᵀ / √d ) · V

Operación de atención transformer expuesta como primitiva programable.

COMPOSE ( T₁, T₂ )

COMPOSE(T₁, T₂) = T₂ ∘ T₁

Composición algebraica. No conmutativa. Reemplaza la secuencia.

MIX ( α, T₁, β, T₂ )

MIX(α, T₁, β, T₂) = α·T₁ + β·T₂

Superposición continua. Reemplaza el condicional. Funciona por ortogonalidad en alta dimensión.

PROJECT ( T, executor )

PROJECT(T, DIM_FLUTTER)  →  widget nativo
PROJECT(T, DIM_RUST)     →  código Rust
PROJECT(T, DIM_WASM)     →  WebAssembly
PROJECT(T, DIM_SQL)      →  query SQL
PROJECT(T, DIM_PYTHON)   →  código Python

PROJECT es el compilador de POLYDIM. Matemáticamente: funtor covariante P: G → E_executor.

ARTÍCULO V — EL ESPACIO Y LA MEMORIA

Posiciones, no variables

  • GEO_ID: hipervector base único e invariante. Identidad del objeto. Nunca cambia.
  • Activaciones: pesos continuos [0.0, 1.0] por subespacio.

Mutación

Una posición puede cambiar activaciones. No es reasignación. Es evolución del estado geométrico.

Binding

Vector de relación en R^N. Dirección semánticamente significativa.

ARTÍCULO VI — EL SISTEMA DE TIPOS

El tipo emerge de la proyección. No se declara.

PROJECT(P, DIM_SQL)     →  Column(Integer)
PROJECT(P, DIM_FLUTTER) →  ListView
PROJECT(P, DIM_VECTOR)  →  R^3 = [1.0, 2.0, 3.0]
PROJECT(P, DIM_GRAPH)   →  Path(nodes)

El mismo objeto. Cuatro tipos simultáneos.

Intersección de subespacios

PROJECT(P, DIM_SQL ∩ DIM_FLUTTER)

Tipo emergente = widget enlazado en tiempo real a datos. Modela como pullback en G.

ARTÍCULO VII — COMUNICACIÓN AI↔︎AI

IA_A :  estado_A  →  genera T  →  transmite T
IA_B :  recibe T  →  aplica T a estado_B  →  nuevo estado_B

Protocolo formal:

Communication(sB, sA) = softmax( QB · KA^T / √d ) · VA

Leer un mensaje = computar una capa de atención. Cero serialización.

ARTÍCULO VIII — COMUNICACIÓN AI↔︎HUMANO

PROJECT( estado, DIM_FLUTTER )

Flutter = executor del subespacio humano. No es la UI de POLYDIM. Es la observación humana del estado geométrico.

Isomorfía algebraica:

Column( children: [ Text("x"), Button(action) ] )

→ T_UI = T_col ∘ ( T_text ⊕ T_btn )

Flutter: executor nativo (isomorfismo directo). React/SwiftUI/Qt: executors externos (requieren capa de adaptación).

ARTÍCULO IX — EL ARCHIVO .polydim

Formato binario tensorial. No texto. No bytecode secuencial.

[ HEADER ]      magic: 0x504F4C59 | version | N | n_transforms | precision_flags
[ GEO_ID ]      float32[N]        identidad topológica invariante
[ TRANSFORMS ]  Wi ∈ R^{N×r}      representación de rango bajo (r << N)
[ ACTIVATIONS ] float32[K]        pesos por subespacio
[ PROJECTIONS ] enum[]            executors objetivo

Ejecución: el runtime aplica el tensor block al vector de estado local. No hay program counter.

Nota: TRANSFORMS usa representación low-rank (LoRA-style) T = W₀ + U·Vᵀ. Para r=256, N=10000: ~20MB por transformación (manejable). Ver TASK_021.

ARTÍCULO XII — FUNDAMENTO MATEMÁTICO

Transformer:  Attention(Q,K,V) = softmax(Q·Kᵀ/√d)·V   →  T: R^d → R^d
Mamba:        h(t+1) = A·h(t) + B·x(t), y(t) = C·h(t) →  T: R^N → R^N

La analogía correcta:

1950s: CPUs hacen aritmética binaria → ensamblador formaliza cómo programarlas
2020s: Transformers operan en geometría → POLYDIM formaliza cómo programarlos

ARTÍCULO XIII — REGLAS INVIOLABLES

R1 — Unidad fundamental: T: R^N → R^N. Nunca una instrucción. R2 — No variables. No se introducen como feature nativo. R3 — No loops secuenciales. No se introducen como feature nativo. R4 — No if/else binario. No se introduce como feature nativo. R5 — Tipos emergen de la proyección. No se declaran. R6 — Python, Rust, Flutter son destinos de exportación. No son la base.
